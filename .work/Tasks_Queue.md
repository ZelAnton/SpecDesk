### [T-071] Реализовать сравнение с параллельными PR, затрагивающими открытый файл (PoC-7) — статус: не начата
По роадмапу PoC-7 (docs/ROADMAP.md) и docs/design/07-review-experience.md Part C: при
редактировании спецификации показывать открытые PR, затрагивающие тот же файл, и сравнивать
выбранный PR — в rendered и raw представлении — с локальной рабочей копией или с `main`.
Протокол уже резервирует `pr.forFile` и `pr.compare.request` (docs/design/09-ipc-protocol.md),
но они не реализованы. Дифф-движок (`SpecDesk.Diff`) принимает произвольную пару (base, head) —
новых алгоритмов не нужно. Область: `SpecDesk.GitHub` (запрос открытых PR по затронутому файлу
и получение содержимого их head), `src/SpecDesk.Host/HostController.Review.cs` /
`.PullRequest.cs`, webview-поверхность сравнения (`webview/src/review/`) и точка входа из
предупреждения о параллельной работе. Ценность: «мягкая блокировка» становится действенной —
автор видит и понимает параллельные изменения, не покидая приложения.
Предпосылки: T-050

### [T-073] Добавить действие «Опубликовать»: merge одобренного PR с гейтом allow-author-publish — статус: не начата
Машина жизненного цикла уже определяет переход Approved → Publish → Published
(`src/SpecDesk.Core/Lifecycle.fs`), но в UI действия «Publish» не существует, а протокольная
строка `action.publish` помечена в docs/design/09-ipc-protocol.md как «not yet built (PoC-10)».
Реализовать срез публикации: команда webview (например `doc.publish`) → host сливает одобренный
PR через GitHub API, удаляет ветку черновика после слияния и обновляет статус на Published;
доступность действия гейтится конфигом `allow-author-publish` из `.spectool.toml`
(docs/design/10-repo-config.md; `src/SpecDesk.Core/WorkflowConfig.fs`). Только простоязычные
формулировки (§8 словаря жизненного цикла). Область: `SpecDesk.GitHub` (merge PR),
`src/SpecDesk.Host/HostController.Review.cs`, lifecycle-chrome webview, контрактные фикстуры.
Ценность: замыкает авторский цикл Edit → … → Publish, который сегодня обрывается на Approved.

### [T-074] Построить простоязычное примирение конфликтов без git-маркеров (PoC-10) — статус: не начата
По PoC-10 (docs/ROADMAP.md; docs/design/04-git-workflow.md; docs/borrowings-from-knowledge.md §C)
приложение должно перехватывать конфликт при отправке/обновлении ревью и вести автора через
простоязычный диалог «Someone else changed this too» (Keep mine / Keep theirs / Combine /
Ask for help) — никогда не показывая git-маркеры. Сегодня конфликтный путь не построен:
rebase-on-send/update отсутствует, конкурирующее изменение закончится сырой ошибкой. Область:
`SpecDesk.Git` (rebase, обнаружение конфликтующих файлов и извлечение обеих сторон),
`src/SpecDesk.Host/HostController.Review.cs` / `.Session.cs`, новый диалог webview (по
компонентному языку дизайн-концепта). Индикаторы «известных конфликтов» в панели репозиториев
уже есть — этот срез даёт путь их разрешения для активного документа. По дизайну это «the
genuinely dangerous part» продукта, её стоит строить отдельным целостным срезом.

### [T-075] Добавить создание новой спецификации из Start-страницы и навигатора — статус: не начата
Приложение открывает, редактирует и удаляет Markdown-файлы, но не умеет создавать новые: в
протоколе нет вида `file.create`/`doc.create`, Start-страница и навигатор предлагают только
открытие существующего. Автор-менеджер не может завести новую спецификацию, не выходя в
Проводник. Добавить «Новая спецификация»: из Start-страницы и из контекстного меню папки в
навигаторе; запрос имени (инлайн-строка по компонентному языку концепта), генерация имени
файла слагом (`src/SpecDesk.Core/Slug.fs`), стартовое содержимое с заголовком, немедленное
открытие созданного файла и появление его в дереве. Область:
`src/SpecDesk.Host/HostController.Session.cs` / `.Workspace.cs`, протокол + контрактные
фикстуры, `webview/src/workspace/tools/` (home-view, file-tree, navigator). Переименование и
удаление как ревьюируемые изменения со сшивкой ссылок — отдельная работа PoC-10, сюда не входит.

### [T-076] Автоматическая фоновая синхронизация зарегистрированных локальных копий — статус: не начата
Ручные «Get updates» / «Share changes» удалены «in preparation for automatic synchronization»
(CHANGELOG.md), остался один ручной Refresh на панели репозиториев; роадмап PoC-4 планировал
background auto-fetch. Построить автоматическую фоновую синхронизацию зарегистрированных
локальных копий: периодическая и focus-gated проверка upstream (fetch), безопасный fast-forward
чистой основной линии (никогда не трогая линии с локальной работой и черновиками), обновление
уже существующих индикаторов «есть обновления»/«конфликт» без действий пользователя. Учесть
троттлинг, офлайн-состояние и отменяемость при отключении аккаунта. Область:
`src/SpecDesk.Host/HostController.Workspace.cs`, `SpecDesk.Git`,
`webview/src/workspace/tools/repositories-panel.ts`. Ценность: авторы всегда видят актуальное
состояние без ручных действий — по замыслу продукта git невидим.

### [T-077] Сохранять пользовательские предпочтения (тема, перенос, режим, окно) между запусками — статус: не начата
Пользовательские предпочтения не переживают перезапуск: тема каждый раз берётся из OS-схемы
(комментарий в `webview/src/index.ts` прямо фиксирует «Persistence across app restarts is out
of scope for this pass»), перенос строк (Wrap) сбрасывается, окно всегда стартует 1280×800 по
центру (`src/SpecDesk.Host/Program.cs`). При этом раскладка доков уже персистится (localStorage,
`webview/src/workspace/dock-store.ts`), а recents/favorites хранит host-сторонний
`WorkspaceStore`. Ввести единое персистентное хранилище предпочтений на стороне host
(JSON-sidecar в AppPaths по образцу `src/SpecDesk.Host/WorkspaceStore.cs`, атомарная запись) и
IPC для чтения/записи; покрыть как минимум тему, перенос строк, режим просмотра и
позицию/размер/максимизацию окна. Ценность: базовая полировка ежедневного UX и готовое место
для будущих настроек (проверка орфографии, поведение ассистента и т.п.).

### [T-078] Добавить поиск по всем спецификациям рабочей области — статус: не начата
Поиск в титульной строке находит только первое совпадение в открытом документе
(`webview/src/index.ts`, обработчик toolbar-search). Для менеджера с десятками спецификаций
нужен поиск по всей рабочей области: host-сторонний поиск по Markdown-файлам корня рабочей
области (ограниченный по объёму и времени, со сниппетами: файл, строка, фрагмент), новый вид
сообщений запрос/ответ, панель результатов в UI (левый рейл или центральное представление),
клик по результату открывает документ на совпадении. Дизайн ассистента
(docs/design/08-ai-agent.md) также предполагает инструмент searchSpec — host-сторонний поиск
станет его основой. Область: новый слайс в `src/SpecDesk.Host/`, протокол + контрактные
фикстуры, `webview/src/workspace/`.

### [T-079] Дать ассистенту read-only контекст документа и AI-подсказки заметок версий и текстов PR — статус: не начата
Ассистент сегодня получает контекст только через явные вложения (chat.attachment.pick) и не
знает об открытом документе; заметки версий и тексты PR генерируются детерминированными
шаблонами (`src/SpecDesk.Host/WorkflowSeeds.cs`). По docs/design/08-ai-agent.md следующий шаг:
дать агенту read-only инструменты getCurrentDoc / getDiff (в hardened-режиме SDK с явным
allowlist, немутирующие), и заменить детерминированные подсказки на AI-подсказки
suggestVersionNote / suggestPrDescription в диалогах «Save a version» и «Send for review» — с
обязательным откатом на существующие шаблоны, когда провайдер недоступен или падает (базовый
рабочий цикл не должен зависеть от агента). Содержимое документа и вывод инструментов
трактуются как данные, а не инструкции. Область: `src/SpecDesk.Ai`,
`src/SpecDesk.Host/HostController.Chat.cs` / `.Session.cs`, диалоги webview.

### [T-080] Реализовать гейтованное предложение правки от ассистента (proposeEdit + подтверждение) — статус: не начата
Жёсткое правило дизайна (docs/design/08-ai-agent.md): любое мутирующее действие агента проходит
через то же подтверждение, что и ручное; в протоколе нет `confirm.request`, а инструмента
proposeEdit не существует. Реализовать гейтованное предложение правки: инструмент proposeEdit
ставит предложение (не применяет), host отправляет webview запрос подтверждения с рендером
отличий предложения от текущего текста (переиспользовать существующую дифф-поверхность), автор
редактирует/подтверждает/отклоняет, и только подтверждённая правка применяется через обычный
путь редактирования документа. Область: `src/SpecDesk.Ai`, `src/SpecDesk.Contracts` + протокол
и контрактные фикстуры, `src/SpecDesk.Host/HostController.Chat.cs`, webview (поверхность
подтверждения по компонентному языку концепта). Это открывает агенту безопасный путь к главной
ценности — «подтяни этот раздел» с применением в один клик.
Предпосылки: T-079

### [T-086] Добавить автоматизированный аудит доступности axe-core в Layer 1 e2e — статус: не начата
Дизайн-концепт (§11) требует WCAG AA, полную клавиатурную доступность и aria-атрибуты, но
автоматической проверки доступности нет — только точечные ручные проверки в e2e. Добавить
аудит axe-core (`@axe-core/playwright`, dev-зависимость пакета `e2e/`) как Layer 1-сценарий:
сканировать ключевые поверхности (редактор в трёх режимах, панели рабочей области, диалоги,
документ проверки PR, вход GitHub) в светлой и тёмной теме; падать на нарушениях
serious/critical; допущенные исключения фиксировать явным списком с обоснованием. Это
превращает требование концепта в исполняемый гейт и автоматически страхует каждую новую
поверхность UI.
