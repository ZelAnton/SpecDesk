публикации материализует рабочую копию default-воркспейса из дерева коммита main, откатывая любые
прямые правки шести исторически отслеживаемых `.work/*.md`-файлов (Tasks_Queue.md, Tasks_Done.md,
status.md, Active_Task.md, Checkpoint.md, Review.md), если они не были закоммичены в main ДО этого
вызова. Митигация соблюдена и в четвёртом батче — правки очереди/архива этого батча коммитятся
отдельным коммитом сразу после архивации, до следующего `jj new main` (следующей публикации).

Дополнительное наблюдение (jj-специфичное, оба последних батча): интеграционный воркспейс
(`_integration`), созданный через `jj workspace add -r <base>`, порождает пустой безымянный
коммит без описания в основании цепочки слияний — этот коммит блокирует `jj git push`
(«Won't push commit … since it has no description»), пока его явно не описать (`jj describe -r
<commit> -m "Start integration workspace for batch <B-id>"`) непосредственно перед push. Это
происходит на каждом батче, использующем эту схему создания `_integration`; стоит иметь в виду
как стандартный шаг перед push, а не считать разовой аномалией.
