# Активная задача

Статус: выполнена
Исходная задача: [T-001] Исправить S-01: PushBranch сообщает об успехе при отклонённом push

Оценка сложности: высокая
Рекомендуемый effort: high

`src/SpecDesk.Git/LibGit2DocumentVersioning.cs:229-248`. `PushOptions` задаёт `CredentialsProvider` и
`OnPushTransferProgress`, но не `OnPushStatusError` — LibGit2Sharp сообщает об отклонении конкретного
ref (non-fast-forward, защищённая ветка, отклонивший pre-receive hook) только через этот колбэк, а
`Network.Push` при этом возвращается как обычно. В результате при коллизии одноимённых веток черновика
(имена по умолчанию детерминированы по дате) push на сервере отклоняется, но `PushBranch` возвращает
успех, и `HostController` (:840, :954) сообщает "Отправлено/обновлено на проверку" и продвигает
жизненный цикл, хотя ревьюер так и не получил коммиты. Интерфейсная документация (`GitPublishing.cs:30-31`)
прямо утверждает, что метод бросает исключение при неудаче.

Критерии готовности:
- В `PushOptions`, передаваемых в `Network.Push`, задан `OnPushStatusError` (или иной способ проверить
  результат push) — отклонение любого ref обнаруживается.
- При отклонённом push `PushBranch` выбрасывает исключение, а не возвращается штатно.
- Комментарий в `GitPublishing.cs:30-31` ("throws on failure") соответствует реальному поведению.
- Добавлен тест, воспроизводящий отклонённый push (текущий тест пушит только в свежий локальный bare-репозиторий).
- Запись в `CHANGELOG.md` под `## [Unreleased]`.

# Активная задача

Статус: выполнена
Исходная задача: [T-002] Исправить S-02: не-GitHub push-эндпоинт получает Windows SSO-креды

Оценка сложности: средняя
Рекомендуемый effort: high

`src/SpecDesk.Git/LibGit2DocumentVersioning.cs:238-241`. Колбэк credentials возвращает
`new DefaultCredentials()` для любого URL, кроме `github.com` — это `GIT_CREDENTIAL_DEFAULT` (Windows
Negotiate/NTLM текущего пользователя), что прямо противоречит соседнему комментарию ("gets no
credential"). Если `.git/config` `pushurl` репозитория (полученного, например, как zip/расшаренная
папка) указывает на хост атакующего, "Отправить на проверку" инициирует NTLM-хэндшейк от имени автора —
NetNTLMv2 challenge/response может быть перехвачен или релеен.

Критерии готовности:
- Колбэк credentials возвращает `null` (или бросает исключение) вместо `DefaultCredentials()` для
  не-GitHub URL.
- Push на не-GitHub URL корректно и безопасно завершается неудачей (NTLM-хэндшейк не инициируется).
- Поведение push на GitHub не изменилось.
- Добавлен тест: push на не-GitHub URL безопасно проваливается.
- Комментарий у соответствующего кода приведён в соответствие с реальным поведением.
- Запись в `CHANGELOG.md`.

# Активная задача

Статус: выполнена
Исходная задача: [T-003] Исправить S-03: принудительный checkout в BeginEdit уничтожает автосохранённую работу другого документа

Оценка сложности: высокая
Рекомендуемый effort: high

`src/SpecDesk.Git/LibGit2DocumentVersioning.cs:101-108` совместно с
`src/SpecDesk.Host/HostController.cs:1674-1685`. Принудительный checkout сбрасывает всё рабочее дерево
целиком, а не только редактируемый документ. Сценарий: черновик A, автосохранение пишет
несохранённые-в-версию правки на диск ("Сохранено" в UI, но без "Сохранить версию"); открывается
документ B (хост сбрасывает состояние на "Опубликовано", оставляя ветку A checked out и грязной); клик
"Редактировать" на B → принудительный checkout сбрасывает файл A к его закоммиченному состоянию, а
буфер редактора с текстом A не сохраняется нигде — работа теряется безвозвратно. Это противоречит
обещанию продукта "Сохранено (непрерывно, автоматически)".

Критерии готовности:
- Checkout, инициированный "Редактировать", отказывает или безопасно обрабатывает случай, когда рабочее
  дерево содержит незакоммиченные изменения, принадлежащие другому документу/ветке, чем редактируемый.
- Автосохранённый буфер черновика A не теряется молча при открытии "Редактировать" на документе B.
- Добавлен тест, воспроизводящий сценарий: автосохранение A грязное → открыть B → Редактировать B →
  содержимое A либо сохраняется, либо редактирование безопасно блокируется с понятным сообщением.
- Запись в `CHANGELOG.md`.

# Активная задача

Статус: выполнена
Исходная задача: [T-004] Исправить S-04: запрос app:// с закодированным NUL валит процесс

Оценка сложности: средняя
Рекомендуемый effort: high

`src/SpecDesk.Host/AppAssetResolver.cs:34` декодирует `uri.AbsolutePath` (`%00` → `\0`), а `:51`
передаёт результат в `Path.GetFullPath`, который бросает `ArgumentException` на встроенном null.
`Program.cs:100-123` (`ServeAsset`) перехватывает только `IOException`/`UnauthorizedAccessException`,
несмотря на собственный комментарий "must never let an exception escape into the message pump".
Спецификация с `![x](a%00.png)` при рендере вызывает запрос `app://repo/a%00.png` — исключение проходит
через обратный P/Invoke-колбэк WebView2 и валит процесс. Это DoS из содержимого документа, приходящего
извне (например, из шаренного репозитория).

Критерии готовности:
- `ServeAsset` (или `AppAssetResolver`) перехватывает/предотвращает некорректные символы в пути
  (встроенный NUL и т.п.), а не только `IOException`/`UnauthorizedAccessException`.
- Запрос `app://repo/a%00.png` возвращает безопасный ответ об ошибке, а не валит процесс.
- Добавлен тест на некорректные символы пути и тест "ServeAsset никогда не бросает исключение наружу".
- Запись в `CHANGELOG.md`.

## Критерии выполнения

## Описание
`src/SpecDesk.Host/HostController.cs:603-646` (`RunDiskAutosave`) снимает снимок `(path,text)` под
`_sync`, затем пишет под `_repoGate` без повторной проверки; `OnDiscard` отменяет таймер, но
`Timer.Dispose()` не дожидается уже выполняющегося колбэка, а мониторы .NET не гарантируют порядок
FIFO. Сценарий: вставка изображения держит `_repoGate`; сработавший колбэк автосохранения проходит
проверку состояния и встаёт в очередь на gate; "Отменить" тоже встаёт в очередь и захватывает gate

ветке, хотя UI показывает "Опубликовано".

  синхронно дожидается завершения уже выполняющегося колбэка через `Timer.Dispose(WaitHandle)` до того,
  как продолжится "Отменить".
- Воспроизведённый сценарий (автосохранение "в полёте" во время "Отменить") больше не оставляет
  отменённое содержимое как грязную рабочую копию на опубликованной ветке.

Рекомендуемый effort: high

- "Отменить", гонящееся с только что открывшимся PR (в момент завершения "Отправить на проверку"),

Исходная задача: [T-007] Исправить S-08: семантический diff не видит правки чекбоксов, сносок и списков определений

Оценка сложности: высокая
Рекомендуемый effort: high

`src/SpecDesk.Markdown/Projection.fs:14-30, 41-66`. `Pipeline.fs` включает `UseTaskLists`,
`UseFootnotes`, `UseDefinitionLists`, но инлайны `TaskList`, `FootnoteLink`/`FootnoteGroup` и блоки
`DefinitionList` попадают в ветку `| _ -> None` — для них нет отдельных случаев проекции. Из-за этого
переключение `- [ ] x` → `- [x] x` проецируется в одинаковый `ListBlock` → оверлей "Показать изменения"
(единственный потребитель этих данных) сообщает "изменений нет" для реальной, отрендеренной правки;
правка тела сноски или термина/определения также невидима для diff.

Критерии готовности:
- `Projection.fs` получает случаи проекции для инлайнов `TaskList`, `FootnoteLink`/`FootnoteGroup` и
  блоков `DefinitionList` (расширение DU `Ast`, либо отображение состояния задачи на элемент списка) —
  вместо падения в `| _ -> None`.
- Переключение чекбокса задачи детектируется diff'ом как `Changed`, а не как отсутствие изменений.
- Правка тела сноски и термина/определения списка определений детектируется diff'ом.
- Добавлены тесты diff на task-list, сноски и списки определений.
- Запись в `CHANGELOG.md`.
