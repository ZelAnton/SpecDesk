1 · Product & principles

Authors come from editing documents in Office; they must never think in git terms. Reviewers are more technical but stay inside the same plain-language world. The tone is calm, trustworthy, document-centric — office software, not a developer IDE.

  1. No git vocabulary. Edit, Saved, Save a version, Send for review, In review, Approved, Publish — never branch/commit/push/PR/merge/conflict.
  2. Document-first. The content is the hero; chrome is quiet and recedes.
  3. Markdown is the single source of truth. The rendered surface reads like a typeset document, reused across preview, formatted mode, diff and comparison.
  4. Calm lifecycle. States are legible at a glance, never alarming.
  5. Progressive disclosure. Rich review / PR / AI capability is available but only what you need is on screen.

2 · Visual direction

Three directions were explored; B — “Desk” was chosen: a structured, lightly-tinted app with docked panels. A — “Paper” is kept as an approved alternate warm light theme; a dark theme is defined alongside. Shared across all: system UI font, serif headings + sans body for the document, one blue accent, Office-balanced density.

7 · Surfaces (screen catalogue)

Built and themeable in the interactive files. The architecture keeps a simple core with everything else in panels you open and close.

Editor

  • Three modes — Code (tight source), Split (source + rendered, scroll-synced), Formatted (typeset page; WYSIWYG with a formatting toolbar).
  • Inline bars — name-a-draft (accent) and describe-your-changes / version note (warning).

A collapsible left rail holds the repository file navigator and the document outline; a collapsible right rail holds the AI assistant. Either can drop to a slim icon strip so the document takes the room.

Review, PR & checks

  • Status transitions — Draft → Draft review → In review, with reversible steps, Approved→Publish, and Close.
  • Review hub — description, reviewers + Request review, labels + Add, Checks (CI) with live status, and activity.
  • Reviewing others — per-line comment threads (GitHub-style) submitted as Approve / Request changes / Comment.
  • Reviews navigator — mine, assigned to me, or open any by link.

Diff & compare

Added & removed lines plus intra-line word changes shown inline in one row; toggles for Inline/Split-lines and Rendered/Source; base selector to compare with published (main). A rendered semantic diff (structural, typeset) is also available.

AI, conflicts, remote & states

  • AI assistant — docked streaming chat; every proposed change passes a confirmation gate (preview → edit → confirm).
  • Remote updates — quiet background check; “Updates available → Pull in / Compare”; a co-editing hint, your draft never overwritten silently.
  • Conflict reconciliation — “Someone else changed this too” with Keep mine / theirs / Combine / Ask for help — understandable differences, never markers.
  • Publish & empty / first-run — gated confirmation; calm empty state with one primary action and recents.

8 · Accessibility & next steps

WCAG AA contrast; full keyboard reach with visible focus rings; aria-pressed / role=radiogroup / aria-live on the relevant controls; state never encoded by colour alone. Icons: Lucide (1.5px stroke, currentColor). Light is baseline; dark fully tokenised.

Next: fold the new PR/CI/navigator patterns into the Design Guide as formal sections; build a clickable prototype of the workspace shell; finalise Settings and the send-for-review composer. Vocabulary for the reviewer/advanced layer (Checks, Compare with published, Check for updates) is plain-language by default and easy to rename.
