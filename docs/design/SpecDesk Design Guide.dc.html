SpecDesk Design Guide

The implementable design system for SpecDesk — a calm, document-first Markdown editor that hides git behind plain language. Built on Direction B (“Desk”): a structured, lightly-tinted app with docked panels. Direction A (“Paper”) ships as an approved alternate warm light theme. A dark theme is defined alongside.

Use the Cool / Warm / Dark switch (top-right) to preview every token live on this page. Print or Save as PDF for a static copy.

1 · Principles that shape the visuals

  1. No git vocabulary, ever. Authors see Edit, Saved, Save a version, Send for review, In review, Approved, Publish — never branch/commit/push/PR/merge/conflict. The lone exception is the plain-language version note.
  2. Document-first. The content is the hero; chrome is quiet and recedes. Tint and shadow are used sparingly to separate panels, never to decorate.
  3. Markdown is the single source of truth. The rendered surface looks like a well-typeset document, not a web page. It is reused across preview, formatted mode, diff and comparison — style it once, beautifully.
  4. Calm lifecycle. State changes are legible at a glance and never alarming. Errors are plain language, never stack traces.
  5. Office-familiar, not IDE. A clear toolbar, version history, margin comments, a review mode — patterns a Word/Docs user recognises.

2 · Foundations — tokens

Everything below is driven by CSS custom properties. The default :root is the cool theme; [data-theme="warm"] and [data-theme="dark"] override only what changes. The accent and all semantic/lifecycle hues are shared across the two light themes.

3 · Rendered-document stylesheet

The product’s core reading surface — reused in preview, formatted mode, diff and comparison. A CSS stylesheet over standard Markdown HTML (h1–h6, p, ul/ol, blockquote, table, pre/code, img, hr, a). Serif headings, sans body, generous measure (max ~68ch in formatted page mode).

4 · Editor theme (CodeMirror 6)

A quiet Markdown theme on surface. Headings take the accent at weight 600; list/quote markers are muted; everything else is body ink. Active line is a faint accent-soft; selection is accent at low alpha.

5 · Components

5.1  Buttons

primary · secondary · ghost · danger · disabled

Height 30px (6px×13–14px padding), radius 6. Primary = accent fill, one per context. Secondary = surface + strong border. Ghost = muted text, transparent. Hover deepens fill/adds surface-raised; active uses accent-active; focus shows a 2px --focus-ring outline at 2px offset.

5.2  Segmented control & toggle

Wrap

View-mode switch is the filled segmented control (Direction B): sunken track, active segment a white card with shadow-card. Carries role="radiogroup" / aria-checked. Binary options (Wrap) use a switch.

5.3  Inputs & the inline prompt bars

Two tinted strips appear directly under the toolbar. Both use a soft semantic tint, a clear label, an input, and a primary + ghost action. They are the only places a “commit-like” idea surfaces — always in plain words.

Name this draft
Describe your changes
Reordered order-states and added the refund-timing note ▾

Draft-name bar = accent-soft (it’s a beginning); version-note bar = warn tint (a deliberate checkpoint). The ▾ expands the note to a multi-line textarea.

Every mutating action the assistant proposes is gated: it renders a preview card (the change, in plain language), the author can Edit, and nothing happens until Confirm. Assistant messages are quiet (no bubble fill); user messages get a subtle surface-sunken bubble. Streaming text uses a soft caret, no spinner.

7 · Layout patterns

Single resizable window, target ≥ 1024×700. Vertically: toolbar → optional inline prompt bar → content area (1–2 panes) → optional docked side panel.

Splitters are 1px --border, widening to a 4px grab zone with an --accent-soft hover. Docked vs overlay: Review & AI dock as a right column (380px, resizable) on wide windows; below ~1200px they become a right overlay drawer with a scrim. Comments live in the document margin (formatted) or a gutter rail (source), not the dock. Only one heavy panel docked at a time.

8 · Interaction states

  • Hover — surfaces lift to --surface-raised; primary deepens to --accent-hover (120ms).
  • Focus — always visible: 2px --focus-ring outline, 2px offset. Never remove outlines.
  • Active / disabled — active = --accent-active; disabled = sunken fill, faint text, not-allowed, 100% opacity (no ghosting).
  • Loading — skeleton blocks in --surface-sunken with a slow shimmer; background sync = a quiet text label (“Syncing…”), never a modal.
  • Empty — centred, calm: a one-line prompt + a single primary action (“Open a spec”). No illustrations-as-noise.
  • Error / notice — plain language, inline where possible; toasts bottom-right on --surface with a left semantic rule, auto-dismiss for info, manual for errors.

9 · Accessibility & iconography

  • Body and UI text meet WCAG AA (4.5:1); large text and chips ≥ 3:1. The soft semantic fills are paired only with their darker -fg token, which is AA on that fill.
  • Full keyboard reach; logical focus order; aria-pressed on toggles, role="radiogroup" on the segmented control, aria-live="polite" on the status region.
  • Never encode state by colour alone — every lifecycle chip pairs colour with a word (and a dot).

Icons: use Lucide (MIT, bundled SVG sprite) — 1.5px stroke, 16/18/20px, currentColor. It is calm, complete, and matches the line weight of the UI. Toolbar actions may be icon+label or label-only; never icon-only without a tooltip + aria-label.
